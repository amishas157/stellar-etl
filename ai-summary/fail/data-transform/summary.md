## Meta-Patterns

1. **XDR Field Verification First**: Multiple hypotheses (003, 004, 005) failed because the alleged source XDR field does not exist in the Stellar protocol. Always inspect the XDR struct definition before proposing a path mix-up.

2. **Cross-Check Published Findings**: Investigations 007, 008, 009, and 010 all re-investigated already-confirmed and published findings. Check `ai-summary/success/` before submitting to avoid duplicate work.

3. **Repo Tests as Design Intent**: When the repository's own unit tests and golden outputs lock in a specific representation (014, 015, 017), that is authoritative evidence of design intent. A PoC demonstrating format inequality is insufficient without a codebase contract requiring the other format.

4. **Discriminator Column Recoverability**: A column that collapses to `0` for inapplicable rows is not unrecoverable data corruption if a sibling discriminator column (e.g., `trade_type`) lets downstream consumers recover the original semantic (016).

5. **SDK Abstraction Layer**: SDK helpers such as `Envelope.SourceAccount()` already handle fee-bump and multi-source disambiguation. Always verify SDK method semantics before assuming that direct XDR field access produces a different result (006).

6. **Intentional Legacy Semantics**: Some exported fields carry historical BigQuery schema conventions that differ from what their name implies (002). Golden output files and checked-in tests are the authoritative source of truth for intended values.

7. **Upstream Model Omissions Are by Design**: When the upstream ingest SDK intentionally omits a field for a certain asset class (013, custom-token `Asset`), the ETL leaving that column empty is correct behavior, not a gap.
