## Meta-Patterns

1. **XDR Field Verification First**: Multiple hypotheses (003, 004, 005) failed because the alleged source XDR field does not exist in the Stellar protocol. Always inspect the XDR struct definition before proposing a path mix-up.

2. **Cross-Check Published Findings (All Subsystems)**: Investigations 007–010, 024a, 025, 026, 027, and 028 all re-investigated already-confirmed and published findings. Always check `ai-summary/success/` across **all subsystems** — not just the target subsystem — before submitting; the same underlying bug may be published under a different classification (e.g., `data-integrity` vs `data-transform` vs `export-pipeline`).

3. **Repo Tests as Design Intent**: When the repository's own unit tests and golden outputs lock in a specific representation (014, 015, 017), that is authoritative evidence of design intent. A PoC demonstrating format inequality is insufficient without a codebase contract requiring the other format.

4. **Discriminator Column Recoverability**: A column that collapses to `0` for inapplicable rows is not unrecoverable data corruption if a sibling discriminator column (e.g., `trade_type`) lets downstream consumers recover the original semantic (016).

5. **SDK Abstraction Layer**: SDK helpers such as `Envelope.SourceAccount()` already handle fee-bump and multi-source disambiguation. Always verify SDK method semantics before assuming that direct XDR field access produces a different result (006).

6. **Intentional Legacy Semantics**: Some exported fields carry historical BigQuery schema conventions that differ from what their name implies (002). Golden output files and checked-in tests are the authoritative source of truth for intended values.

7. **Upstream Model Omissions Are by Design**: When the upstream ingest SDK intentionally omits a field for a certain asset class (013, custom-token `Asset`), the ETL leaving that column empty is correct behavior, not a gap.

8. **Failed-Operation Defaults Match Upstream SDK Convention**: Zero-default outcome amounts for failed path payments (030/032) and native-asset defaults for failed LP operations (031) exactly mirror the canonical `stellar/go` SDK behavior. Before labeling a default value as "fabricated," trace the equivalent upstream SDK processor to confirm whether the pattern is ecosystem-wide convention.

9. **Float64 Schema Design Limit vs. Conversion Bug**: Precision loss through `ConvertStroopValueToReal()` / `big.NewRat().Float64()` is an inherent property of the float64 schema type — the best possible conversion available. This is NOT_VIABLE unless the code either (a) uses an inferior inline conversion when the shared helper was available, or (b) computes an exact string and then unnecessarily round-trips it through `ParseFloat`. Investigations 034–038 all fell into category (c): no better conversion path existed but was unused.

10. **Dead Code Is Not a Better Path**: `transactionOperationWrapper.Details()` is never called anywhere in the codebase (confirmed via grep). Citing dead code as evidence of "a precision-preserving alternative that was not used" does not satisfy the VIABLE criterion. Always verify a cited alternative is reachable before treating it as proof of a missed opportunity.
