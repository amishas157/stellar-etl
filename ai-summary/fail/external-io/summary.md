## Meta-Patterns

1. **Latent Code ≠ Active Bug**: Several hypotheses identified genuine logic flaws in `GetAllHistory()` and `UpdateOrderbook()`, but neither code path is wired to a live export command. Latent bugs should be noted but are out of scope for "current export correctness" investigations.

2. **SDK Contracts Already Handle It**: Multiple hypotheses assumed raw XDR wrapper types are opaque, but `xdr.TransactionEnvelope.Operations()` and related helpers already normalize fee-bump envelopes. Always audit SDK envelope helpers before assuming a wrapper type is transparent only to manual iteration.

3. **Intentional Schema / By-Design Behavior**: The most common rejection reason. Fee-bump field splits (`max_fee` vs `new_max_fee`), claimable-balance Parquet omission, `create_passive_sell_offer` field omission, `export_assets` one-asset-per-operation model, `details_json` alias, and per-ledger batch compaction are all documented (or consistently coded) design decisions — not bugs. Check code comments, README, and Horizon's contract before filing.

4. **Bare JSON Numbers in Generic Details Maps**: The `Details map[string]interface{}` pattern (used in trade effects, operation details, and sequence-bump effects) stores `uint64`/`int64` values as bare JSON numbers. This is by-design per the published schema in `internal/transform/schema.go` and reinforced by existing tests. Precision loss is a downstream consumer responsibility, not an ETL-side bug. Do not re-investigate this pattern.

5. **Cross-Subsystem Duplicates**: Three investigations (009, 025, 026) re-discovered findings already confirmed in `data-transform` or `data-input` subsystems. Always check success and fail directories across all subsystems before filing. The `--captive-core` zero-LCM pattern and the `--limit` counting-at-wrong-granularity pattern are both fully documented in `data-input/` successes.

6. **Live Trigger Required for JSON Number Precision**: Hypotheses about `uint64`/`int64` fields that could theoretically exceed `2^53` (e.g., `offer_id`) are insufficient without evidence that real-world values currently cross the IEEE-754 safe integer boundary. XDR field width alone does not establish a live bug.

7. **Parquet-Specific Traps**: (a) An empty resource batch produces a valid empty Parquet file — nil schema is only dangerous with real rows. (b) `details_json` is a JSON alias, not a distinct Parquet column. (c) Config-setting window fields mirror their JSON source identically before conversion. Verify upstream struct field values before treating Parquet mapping anomalies as data loss.
