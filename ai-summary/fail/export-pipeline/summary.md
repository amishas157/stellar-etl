## Meta-Patterns

1. **Protocol 23 backward-compatibility aliasing**: When XDR renames a field (e.g. `BucketList*` → `SorobanState*`, `TotalByteSizeOfBucketList` → `TotalByteSizeOfLiveSorobanState`, `BucketListSizeWindowSampleSize` → `LiveSorobanStateSizeWindowSampleSize`), both old and new column names are intentionally populated from the renamed field to preserve downstream BigQuery compatibility. This applies to config-setting exports, ledger exports, and potentially other entity types. Always check git history across protocol boundary commits and the current generated XDR before concluding that two columns sharing a source value is a bug.

2. **Dormant / unreachable code paths**: Several helpers (`GetAllHistory()`, `UpdateOrderbook()`, `transactionOperationWrapper.Details()`) contain real bugs (inverted loop condition, Go slice value-semantics, spelling errors) but are not wired into any active CLI export command. These should be tracked for re-opening when a caller is added, but they do not constitute live data-corruption hypotheses.

3. **Upstream-mirroring design**: The ETL intentionally mirrors the behavior of `stellar/go` ingest processors and Horizon's history DB conventions, including legacy quirks (`transaction_count` = successful only, `authorized_to_maintain_liabilites` without `_flag`, muxed address flattening in trades, `set_authorized` SAC events skipped as SDK TODO). Before filing a divergence as a local bug, verify against the upstream processor and Horizon API definition.

4. **No XDR source, no viable finding**: A schema column that has no corresponding field in current generated XDR (e.g. `auto_bump_ledgers`) or whose only realistic source is the same field as another column (e.g. `soroban_resources_read_bytes`) is dead/legacy schema, not an active export bug, unless a distinct on-chain replacement value can be established.

5. **Uniform Parquet null-to-zero pattern**: The entire codebase converts `guregu/null` types to primitive zero values in Parquet output. This is an explicit architectural decision (zero `OPTIONAL` Parquet tags codebase-wide). Entity-specific "null collapse" hypotheses will always be NOT_VIABLE under this design — even when a sibling discriminator column (e.g. `trade_type`) makes the null semantics recoverable, it does not override the architectural decision.

6. **Cross-subsystem duplicate checking**: The same code path can be filed under different subsystem labels (`cli-commands`, `export-pipeline`, `utilities`, `data-transform`, `data-integrity`). Before filing a hypothesis, check fail and success directories across all subsystems. Restored-key parquet gap, claimable-balance parquet skip, and eviction-iterator empty shell were each filed multiple times across subsystem boundaries.

7. **Intentional omissions are design decisions**: Code comments that explicitly document "field X is not preserved" or "this is not needed in the current scope" are design decisions, not bugs — even when the omitted data would be useful to downstream consumers.

8. **`PrintTransformStats` input-granularity convention**: All 10 export commands call `PrintTransformStats(len(inputSlice), numFailures)` where `inputSlice` is the pre-transform input items — never the emitted output rows. Fan-out commands (effects, trades, contract_events, token_transfers) and dedup commands (assets) all follow this identical pattern. A separate `bytes written` log line provides the output-level signal. Hypotheses claiming the stats undercount emitted rows will always fail because the function is intentionally transform-operation-oriented at input granularity.

9. **Discriminator-recoverable null collapse is still NOT_VIABLE**: Even when a Parquet field's null-to-zero collapse can be undone by reading another column on the same row (e.g. `trade_type=1` implies pool-related fields are inapplicable), the hypothesis fails. Recoverability does not change the architectural decision to encode nulls as zeros; it merely proves the collapse is not information-destroying.

10. **Raw-dump XDR level selection**: `export_ledger_transaction` is a raw XDR dump that serializes each `LedgerTransaction` field at its native struct level. When a field appears at a deeper nesting level than a sibling exporter uses, check whether the output schema omits a column that would be lost by drilling down (e.g. the transaction hash is only available inside `TransactionResultPair`). If stripping to the inner type would discard data with no other output column, the higher nesting is intentional.

11. **PoC behavior ≠ PoC of a bug**: A test or PoC that demonstrates what the code does is not proof of a bug unless the output violates an explicitly documented requirement. Commands with documented narrow scope (e.g. `export_assets` as payment-operation-oriented) are not required to handle cases outside that scope, even if extending the behavior would be useful.
