## Meta-Patterns

1. **Unwired code paths**: Dead-end helpers (`GetAllHistory`, `UpdateOrderbook`/`StreamOrderbooks`) can contain real logic bugs but are not actionable findings until connected to a shipped export command. Always confirm reachability before escalating.

2. **Batch-boundary working-as-designed**: `StreamChanges()` intentionally absorbs a single-ledger remainder into the preceding batch (`batchSize+1` edge case). Existing tests codify this behaviour and code comments acknowledge the inclusive-end convention — not a bug.

3. **XDR type constraints rule out error paths**: Several "ignored error" concerns are unreachable because the underlying XDR type restricts the value domain (e.g., unsigned `TimePoint` cannot go negative, `MustV0()` only panics on nil `V0`). Always trace the XDR type and realistic value range before escalating an omitted error check.

4. **Crash vs silent corruption**: A zero-value XDR union struct (e.g., `LedgerCloseMeta{}`) causes a nil-pointer panic via `MustVN()` dereference rather than silently emitting zero-valued fields. Always trace through the SDK accessor chain to determine whether the failure mode is a crash or silent corruption.

5. **Duplicates against confirmed successes**: Three hypotheses (008, 015, 016) independently rediscovered already-confirmed and published success findings. Always check `ai-summary/success/{subsystem}/` and the global success summary before deep-diving into a hypothesis.

6. **Intentional command scope**: `export_assets` is intentionally a narrow payment/manage-sell-operation asset-reference export, not an exhaustive asset inventory. Narrow readers that match the documented scope are not data-loss bugs.

7. **Unsupported modes**: `export_ledger_entry_changes` unbounded/continuous mode is documented as "Currently Unsupported" in the README. A confirmed bug in that mode does not constitute an in-scope data-integrity finding.

8. **Protocol-level guarantees validate code assumptions**: Stellar's strictly monotonic close times (preventing equal-adjacent-timestamp gaps) and second-precision `xdr.TimePoint` (preventing sub-second boundary confusion) make code that appears fragile actually sound.

9. **Split TOID construction is intentional**: Trade operation ID `+1` in `TransformTrade()` normalises a 0-based input TOID to the 1-based history operation ID contract. When TOID construction is split across reader and transformer layers, verify the upstream indexing contract before calling it an off-by-one bug.
