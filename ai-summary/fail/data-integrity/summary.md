## Meta-Patterns

1. **Unwired Orderbook Pipeline**: `internal/input/orderbooks.go` contains multiple plausible data-loss bugs (`UpdateOrderbook` value semantics, zero-row emission on transform error, single-batch receiver). All were rejected because no active CLI command or export path wires this pipeline. Do not investigate orderbook helpers until a live caller is confirmed.

2. **Legacy Config-Setting Aliases**: `TransformConfigSetting()` and `ConfigSettingOutputParquet` contain columns that are intentional backward-compatible aliases over a single XDR source (`DiskRead*` / `LiveSorobanState*`). Apparent duplications or suspicious Parquet variable reuse in this area are expected, not bugs.

3. **SorobanResources Has No Separate ReadBytes**: The Stellar XDR `SorobanResources` type only exposes `DiskReadBytes`—there is no separate `ReadBytes` field. The `soroban_resources_read_bytes` column is a backward-compatible alias. This hypothesis has been rejected six times; do not resubmit.

4. **Intentional JSON/Parquet Schema Divergence**: This codebase deliberately keeps JSON and Parquet schemas non-identical. Several JSON-only convenience fields (`tx_signers`, `details_json`, `*_strkey` variants) have no Parquet counterpart. Field-count mismatch is not a valid corruption oracle; look for semantic value differences, not structural ones.

5. **Upstream Effect Schema Is the Source of Truth**: Effects intentionally mirror `stellar/go` Horizon semantics. Before reporting an effect field mismatch (e.g., `address` vs `details.contract`), verify that the same behavior is present in the upstream `processors/effects/effects.go` and its tests.

6. **Dead Code Before Suspicious Branches**: Several apparently dangerous code paths (`TransformTtl` empty-row return, `AssetFromContractData` `"Native"` case, non-v0 event-body panic) are preceded by guards that make them unreachable for legitimate on-chain data. Always trace guard conditions and caller-side input filtering before classifying a branch as a live bug.
