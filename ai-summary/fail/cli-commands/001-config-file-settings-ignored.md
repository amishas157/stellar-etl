# H001: `--config` loads a file but all command settings still fall back to flag defaults

**Date**: 2026-04-12
**Subsystem**: cli-commands
**Severity**: High
**Impact**: wrong-network / wrong-destination export configuration
**Hypothesis by**: gpt-5.4, high

## Expected Behavior

When a user provides `--config /path/to/stellar-etl.yaml` (or relies on the default `~/.stellar-etl.yaml`), command settings from that file should affect execution the same way explicit CLI flags do. For example, a config that sets `testnet: true`, `output: /tmp/testnet.txt`, or `cloud-provider: gcp` should cause the exporter to read testnet data and write/upload to the configured destinations even when those flags are omitted from the command line.

## Mechanism

`root.go` initializes Viper and even prints `Using config file: ...`, but none of the command flags are bound into Viper and none of the flag readers consult Viper at all. Every export command ultimately reads values only through `flags.GetBool/GetString/GetUint32/GetInt64`, so config-file values are silently ignored and execution proceeds with built-in defaults such as pubnet and `exported_<object>.txt`.

## Trigger

1. Create a config file containing settings such as `testnet: true` and `output: /tmp/configured-transactions.txt`.
2. Run a normal export command with only the required ledger flags, for example `stellar-etl --config cfg.yaml export_transactions -s 1000 -e 1001`.
3. The process prints `Using config file: cfg.yaml`, but the export still follows the default flag values unless the same settings are also passed explicitly on the CLI.

## Target Code

- `cmd/root.go:initConfig:51-74` — reads the config file with Viper and announces it
- `internal/utils/main.go:MustFlags:320-440` — reads command settings only from Cobra flags
- `internal/utils/main.go:MustCommonFlags:460-537` — same flag-only read path for common export settings
- `internal/utils/main.go:MustArchiveFlags:541-562` — same flag-only read path for archive output/limit settings

## Evidence

Repository-wide search shows `viper.ReadInConfig()` and `viper.AutomaticEnv()` in `cmd/root.go`, but no `viper.BindPFlag`, `viper.BindPFlags`, or `viper.Get*` calls anywhere in `cmd/` or `internal/utils/`. That means the config file is loaded and advertised, yet its values never reach any export command.

## Anti-Evidence

The README does not document a config-file schema, so one could argue the feature is under-specified rather than fully promised. But the user-facing `--config` flag plus the unconditional `Using config file:` banner make it reasonable for operators to believe the loaded file is actually steering the export.

### Trace Summary

Traced the complete Viper/Cobra integration path: `root.go:initConfig` calls `viper.SetConfigFile`, `viper.AddConfigPath`, `viper.AutomaticEnv`, and `viper.ReadInConfig`. The config file is loaded and announced but no `viper.BindPFlag`, `viper.BindPFlags`, or `viper.Get*` calls exist anywhere in the codebase. All 10 export commands read settings exclusively through `pflag.FlagSet.Get*()` in `MustFlags`, `MustCommonFlags`, and `MustArchiveFlags`. The Viper integration is indeed dead code.

### Code Paths Examined

- `cmd/root.go:initConfig:52-74` — Viper setup loads config file and prints banner, but values are never consumed
- `internal/utils/main.go:MustFlags:320-441` — reads all settings via `flags.Get*()` (pflag), no Viper calls
- `internal/utils/main.go:MustCommonFlags:460-537` — same pflag-only read path
- `internal/utils/main.go:MustArchiveFlags:541-562` — same pflag-only read path
- `internal/utils/main.go:AddCommonFlags:232-246` — flag definitions, no `viper.BindPFlag` calls
- Repository-wide grep for `viper\.` — only 6 hits, all in `cmd/root.go`, all setup/read/announce, none binding or retrieving values

### Why It Failed

This is **incomplete Cobra scaffolding**, not a data correctness bug. The `root.go` Viper boilerplate was generated by `cobra init` and never completed — no config schema was designed, documented, or integrated. The actual design contract is CLI-flag-only configuration, which is the documented and tested interface across all 10 export commands and the README.

For this to cause silent data corruption, a user would need to: (1) create a config file with guessed key names matching undocumented Viper conventions, (2) rely on that config without testing the output, (3) never pass the equivalent CLI flags. The README documents all settings exclusively as CLI flags and provides no config file schema. No config-file-driven workflow is documented or supported.

The `--config` flag and "Using config file:" message are misleading UX artifacts from scaffolding, but the data export pipeline itself produces correct output through its documented CLI flag interface. This is dead code / incomplete feature, not a data integrity violation.

### Lesson Learned

Cobra scaffolding (`cobra init`) generates Viper config boilerplate by default. The presence of `--config` flag and `viper.ReadInConfig()` does not mean config file support was intentionally designed — check whether any code actually reads from Viper via `BindPFlag` or `Get*`. Unused scaffolding is a UX/documentation concern, not a data correctness bug in the export pipeline.
