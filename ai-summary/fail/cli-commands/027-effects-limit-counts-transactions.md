# H003: `export_effects --limit` caps transactions instead of effect rows

**Date**: 2026-04-12
**Subsystem**: cli-commands
**Severity**: High
**Impact**: structural row-count corruption
**Hypothesis by**: gpt-5.4, high

## Expected Behavior

`export_effects --limit N` should emit at most `N` effect rows, because the command advertises the limit as the maximum number of exported effects. A downstream batch that asks for a single effect should not silently receive every effect generated by the first transaction in range.

## Mechanism

`export_effects` forwards `limit` into `input.GetTransactions()`, and that helper stops after collecting `N` transactions, not `N` effects. `transform.TransformEffect()` then walks every operation in each returned transaction and appends every derived effect row; the command writes the entire `[]EffectOutput` slice without any row-level truncation.

As a result, the public `effects` limit is really a hidden transaction limit. Any first transaction that emits multiple effects will cause `--limit 1` to over-export while still logging normal stats and success.

## Trigger

Run `export_effects -l 1` over a range whose first returned transaction contains a multi-effect operation, such as an `AllowTrust`, `SetTrustLineFlags`, sponsorship, or payment path that emits more than one Horizon-style effect. The checked-in one-ledger effect fixture already shows this fan-out behavior: ledger `30820015` contains repeated `operation_id` values (`132370956487307265`, `132370956487307274`, `132370956487426049`), proving that a single operation can legitimately expand into multiple exported effect rows.

## Target Code

- `internal/utils/main.go:AddArchiveFlags:250-254` — public flag contract says `limit` is the maximum number of exported `effects`
- `cmd/export_effects.go:21-27` — passes the user limit into `GetTransactions()`
- `cmd/export_effects.go:34-56` — writes every effect row returned from `TransformEffect()`
- `internal/input/transactions.go:GetTransactions:22-70` — enforces `limit` on transactions, not effects
- `internal/transform/effects.go:23-50` — transforms one transaction into `[]EffectOutput`
- `internal/transform/effects.go:53-168` — appends multiple effects per operation/transaction
- `cmd/export_effects_test.go:22-25` — existing `"range too large"` case exercises the public limit flag on this command

## Evidence

The command's own comment block says `limit` is the maximum number of effects to export, but the only limit check is inside `GetTransactions()`. The checked-in effect fixture for a single ledger has seven rows while only four distinct `operation_id` groups appear, and three of those groups occur twice, confirming that one transaction can generate more effect rows than the number of transactions read.

## Anti-Evidence

`PrintTransformStats(len(transactions), numFailures)` reveals that the implementation internally thinks in transactions, not effects. But that internal bookkeeping does not rescue the user-facing contract: the CLI flag and command description are framed around exported effects, so silently substituting a transaction cap still produces wrong output cardinality.

### Trace Summary

Traced the limit enforcement path from `AddArchiveFlags("effects", ...)` through `GetTransactions()` (transactions.go:51,65) and the export loop in `export_effects.go:34-56`. Confirmed the limit is applied at the transaction level, not the effect level. This is the identical pattern already investigated and rejected in fail/025 (assets limit overruns first ledger) and fail/026 (token-transfer limit counts ledgers). The fail/026 review explicitly calls out `export_effects` as having this same granularity mismatch.

### Code Paths Examined

- `cmd/export_effects.go:21-27` — passes limit to `GetTransactions()`, which enforces it on transaction count
- `cmd/export_effects.go:34-56` — iterates all effects from each transaction, no second-stage row cap
- `cmd/export_effects.go:85` — init comment says "limit: maximum number of effects to export" but implementation limits transactions
- `internal/input/transactions.go:51` — `for int64(len(txSlice)) < limit || limit < 0` — limit on transactions
- `internal/input/transactions.go:65` — `if int64(len(txSlice)) >= limit && limit >= 0 { break }` — ledger-boundary break also on transaction count

### Why It Failed

This is the same limit-granularity-mismatch pattern already rejected twice in this subsystem:

1. **Duplicate of established pattern.** Fail/026 explicitly identifies `export_effects` as having "the identical granularity mismatch" and concludes: "The `AddArchiveFlags` auto-generated limit text creates a misleading contract for ALL multi-row-expansion commands (`export_token_transfer`, `export_effects`, `export_assets`). This is a systemic documentation issue, not a per-command data correctness bug."
2. **All output row values are correct.** Every effect row contains the right address, type, type_string, operation_id, and details. No field holds a wrong value.
3. **Extra correct data is not corruption.** Downstream systems process variable row counts without issue. Receiving more correct effect rows than the limit specified does not cause wrong calculations, hash collisions, or compliance violations.
4. **Systemic documentation imprecision.** `AddArchiveFlags` auto-generates flag text using the object name for every command, creating misleading text whenever the input granularity (ledgers, transactions) differs from the output granularity (effects, token transfers, assets). This is a documentation concern, not silent data corruption.

### Lesson Learned

The limit-granularity-mismatch pattern for multi-row-expansion commands (`export_effects`, `export_token_transfer`, `export_assets`) has been thoroughly investigated. All three instances share the same root cause (`AddArchiveFlags` auto-generated text vs. actual limit enforcement granularity) and the same verdict: documentation imprecision, not data correctness bug. Future hypotheses about `--limit` semantics in any remaining export command should be checked against this established pattern before submission.
