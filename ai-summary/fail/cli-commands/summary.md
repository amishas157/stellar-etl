## Meta-Patterns

1. **Success-directory blindness**: The most common failure mode is re-investigating a bug that has already been confirmed and published. Before generating a hypothesis, check the `ai-summary/success/` directory across **all** subsystems, not just the current one, since transform-layer bugs (parquet converters, field mappings) are shared across subsystem boundaries.

2. **Cross-subsystem scope confusion**: Several cli-commands hypotheses target code in `internal/transform/` (e.g., `getTxSigners`, `ToParquet()` converters). Findings about that shared layer are filed under `data-transform` or `external-io`; always search those subsystem success/fail directories too.

3. **Explicit-skip ≠ missing-case**: A type-switch arm that sets `skip=true` with an explanatory comment is a deliberate scope boundary, not an accidental fall-through. Only treat a missing case as a bug if `skip` remains false and execution continues into a write call.

4. **Parquet nil-schema safety**: parquet-go v1.6.2 handles `NewParquetWriter(file, nil, 1)` + no written rows gracefully, producing a valid empty PAR1 file. Do not assume a nil schema argument causes a panic without tracing the library's own nil guards.

5. **XDR source-type verification**: For field-mapping and type-audit hypotheses, the upstream XDR field type is the decisive check. A widened JSON struct field or a suspiciously duplicated assignment is not corruption when the underlying protocol type is narrower or has no distinct counterpart.

6. **Legacy alias columns**: The config-setting and Soroban resources schemas contain columns whose names no longer match distinct protocol fields after protocol evolution. Confirm that a separately named column has a distinct on-chain source before calling aliased values corruption.

7. **Documented-contract asymmetry**: JSON/parquet divergence and JSON-only features (e.g., `--extra-fields`, `details_json`) are not bugs when the flag or feature is explicitly documented as format-specific. Always verify the documented contract before treating a format difference as silent data loss.

8. **Parse-time fatals vs. silent corruption**: A `time.Parse` failure that exits via `Fatal` is a loud UX/docs mismatch, not silent integrity loss. For this objective, a format or validation error only qualifies if execution continues and produces a plausibly wrong result.

9. **Operation-derived semantics**: Commands built on payment/offer operation scans (e.g., `export_assets`) do not claim authoritative lifecycle timestamps. "First seen in range" behavior is consistent with the operation-derived design; a viable bug needs a concrete value contradiction, not just an ambiguous field label.

10. **Overwrite-by-contract**: `MustOutFile` uses `O_TRUNC` by design; no export command promises atomic replacement or append semantics. Output-path truncation is only a viable finding if the command advertises transactional behavior.

11. **Loud panics ≠ silent corruption**: Zero-value XDR dereferences (033, 031) and unsupported-mode setup failures (034) convert apparent "wrong output" bugs into loud crashes. A routing or error-swallowing bug only qualifies for this objective if execution continues past the failure and emits plausibly wrong rows.

12. **Multi-row-expansion limit granularity**: `AddArchiveFlags` auto-generates `--limit` text using the exported object name. For any command that produces multiple rows per input unit (ledger, transaction), the limit applies to input units — not output rows. This is intentional and documented in each command's init comment; golden tests explicitly validate the resulting "over-count." All three affected commands (`export_assets`, `export_token_transfer`, `export_effects`) share the same root cause.

13. **Dead Cobra scaffolding**: `cobra init` generates Viper config boilerplate (`ReadInConfig`, `--config` flag, "Using config file:" banner) by default. The presence of these artifacts does not mean config file support was intentionally designed. Always verify that `viper.BindPFlag` or `viper.Get*` calls actually exist before treating the config path as a functional integration layer.
