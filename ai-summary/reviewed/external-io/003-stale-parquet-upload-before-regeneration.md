# H003: Several export commands can upload stale Parquet files before regenerating them

**Date**: 2026-04-10
**Subsystem**: external-io
**Severity**: High
**Impact**: Non-financial field contains wrong data
**Hypothesis by**: gpt-5.4, high

## Expected Behavior

When a command is run with cloud upload enabled, the object uploaded to GCS should contain the Parquet rows for the current ledger range being exported. Reusing a prior `--output-file` or `--parquet` path should not cause the cloud object to contain data from an earlier run.

## Mechanism

`export_ledgers`, `export_transactions`, `export_operations`, and `export_trades` call `MaybeUpload(..., parquetPath)` before `WriteParquet(...)`. `UploadTo()` opens whatever file already exists at `parquetPath`, copies it to GCS, verifies it, and deletes the local file; only after that does the command write the fresh Parquet file for the current run, so GCS can end up holding a stale but valid previous export.

## Trigger

1. Run one of the affected commands with `--write-parquet`, `--cloud-provider gcp`, and a fixed `parquetPath`.
2. Leave an older Parquet file at that same local path from a previous export.
3. Re-run the command for a different ledger range.  

The GCS object will contain the old rows, while the newly written local Parquet file contains the current range.

## Target Code

- `cmd/export_ledgers.go:70-72` — uploads Parquet before writing it
- `cmd/export_transactions.go:63-65` — uploads Parquet before writing it
- `cmd/export_operations.go:63-65` — uploads Parquet before writing it
- `cmd/export_trades.go:68-70` — uploads Parquet before writing it
- `cmd/upload_to_gcs.go:32-73` — uploads the existing local file and deletes it after success

## Evidence

The affected commands all follow the same inverted order: `MaybeUpload(parquetPath)` then `WriteParquet(...)`. `UploadTo()` does not know whether the file was generated by the current run; it just opens the path, copies bytes to GCS, and removes the local file afterward.

## Anti-Evidence

If the Parquet path does not already exist, the upload fails loudly instead of silently corrupting cloud output. Commands such as `export_assets`, `export_effects`, `export_contract_events`, and `export_ledger_entry_changes` use the correct write-then-upload order.

### Trace Summary

I traced all 8 export commands that use both `WriteParquet` and `MaybeUpload` for Parquet files. Four commands (`export_ledgers`, `export_transactions`, `export_operations`, `export_trades`) call `MaybeUpload` before `WriteParquet`, while four commands (`export_effects`, `export_assets`, `export_contract_events`, `export_ledger_entry_changes`) use the correct `WriteParquet`-then-`MaybeUpload` order. The `UploadTo` function in `upload_to_gcs.go` opens the file at the given path via `os.Open`, streams it to GCS, then deletes the local file — it has no knowledge of whether the file belongs to the current run.

### Findings

The bug manifests in two distinct scenarios:

**Scenario 1 — Crash on first run**: If no pre-existing Parquet file exists at `parquetPath`, `MaybeUpload` → `UploadTo` → `os.Open(path)` fails with file-not-found → `MaybeUpload` calls `cmdLogger.Fatalf` → process terminates. `WriteParquet` never executes. This means the four affected commands **always crash** when both `--write-parquet` and `--cloud-provider gcp` are used for the first time on a clean path.

**Scenario 2 — Stale data upload**: If a Parquet file exists from a prior run (or prior invocation without cloud upload), `MaybeUpload` uploads that stale file to GCS and deletes it locally. Then `WriteParquet` writes the current run's data to a new local file. GCS now contains Parquet data from the wrong ledger range — the previous run's range, not the current one. Downstream consumers (e.g., BigQuery ingestion) receive silently wrong data.

The 4:4 split between correct and incorrect commands is a classic Pattern 3 (export command consistency) bug: the template was copied and someone got the order wrong in half the copies.

### PoC Guidance

- **Test file**: `cmd/export_ledgers_test.go` or a new integration-style test
- **Setup**: Create a mock Parquet file at `parquetPath` containing known sentinel data (e.g., ledger range 100-200). Configure a test that invokes the export command for a different ledger range (300-400) with `--write-parquet` and a mock cloud uploader.
- **Steps**: (1) Write a sentinel Parquet file at the expected parquet output path. (2) Invoke the export command for a different ledger range. (3) Capture what the mock uploader receives.
- **Assertion**: Assert that the data passed to the upload function matches the current ledger range (300-400), not the stale sentinel data (100-200). Under the current code, the assertion will fail because the uploader receives the stale file. Alternatively, for the crash scenario: invoke the command with no pre-existing Parquet file and assert that it completes without `Fatalf`.
